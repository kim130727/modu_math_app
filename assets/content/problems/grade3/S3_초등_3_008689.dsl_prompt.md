## 5-1) Problem-Solving Hint Rule

When enough information is visible, `SEMANTIC_OVERRIDE["domain"]` may include a lightweight `problem_solving` block.

1. understand
2. plan
3. execute
4. review

```python

        "given_refs": [],
        "target_ref": "answer.target",
        "condition_refs": [],
    },
    "plan": {
        "method": "",
        "description": "",
    },
    "execute": {
        "expected_operations": [],
    },
    "review": {
        "check_methods": [],
    },
}
````

* `problem_solving` belongs inside semantic `domain`.
* It is a meaning-level solving hint, not the actual worked solution.

## 5-2) Minimal SOLVABLE Shape

When adding `SOLVABLE`, prefer this shape to strictly comply with `modu.solvable.v1`:

```python
SOLVABLE = {
    "schema": "modu.solvable.v1",
    "problem_id": "...",
    "problem_type": "...",
    "inputs": {
        "total_ticks": 1,
        "target_label": "",
        "target_ticks": 1,
        "target_count": 1,
        "unit": ""
    },
    "given": [],
    "target": {},
    "method": "",
    "plan": [
        "단계별 풀이 과정을 문장으로 작성합니다."
    ],
    "steps": [
        {
            "id": "step.1",
            "operation": "...",
            "expr": "...",
            "value": 1
        }
    ],
    "checks": [
        {
            "id": "check.1",
            "type": "...",
            "expr": "...",
            "expected": 1,
            "actual": 1,
            "pass": True
        }
    ],
    "answer": {
        "value": 1,
        "unit": "",
        "derived_from": "step.1"
    }
}
```

Rules:

* `inputs` is REQUIRED. `total_ticks`, `target_ticks`, `target_count` must be numbers. `unit` must be a string (use `""` instead of `None`).
* `checks` array items MUST contain `id`, `expr`, `expected`, `actual`, and `pass`. Do not omit `expected` or `actual`.
* `answer.value` MUST be a single Number. Do not use dictionaries, lists, or `None`.
* `answer.unit` MUST be a string. Do not use `None`.
* `given` should be derived from semantic domain objects.
* `target` should match `SEMANTIC_OVERRIDE["answer"]["target"]`.
* `method` should match `SEMANTIC_OVERRIDE["domain"]["problem_solving"]["plan"]["method"]` when available.
* `steps` should contain concrete operations and values.
* `answer.value` should be derived from the last relevant step.
* Keep `SOLVABLE` small and verifiable.
* If the solution cannot be derived confidently, leave TODO comments and use conservative numeric values (like `0`) just to pass schema validation.

## 5-3) Semantic Answer vs Solvable Answer

Use this distinction:

* `SEMANTIC_OVERRIDE["answer"]` describes what must be found.
* `SOLVABLE["answer"]` contains the computed result.

Recommended semantic answer shape:

```python
"answer": {

        "type": "",
        "description": "",
    },
    "value": None,
    "unit": "",
}
```

Recommended solvable answer shape:

```python
"answer": {
    "value": None,
    "unit": "",
    "derived_from": "",
}
```

Rules:

* If the answer is printed in the image, semantic `answer.value` may be filled.
* If the answer must be computed, semantic `answer.value` should usually be `None`.
* The computed value should appear in `SOLVABLE["answer"]["value"]`.
* `SOLVABLE["answer"]["derived_from"]` should point to the step that produced the answer when possible.

## 6) Layout Guidance

* Keep labels and points visually separated.
* Keep coordinates simple and stable.

* Layout slots may contain coordinates and visual placement.

## 6-1) Grid Pattern Rule (Geometry Dot Problems)

* When the image clearly includes a dotted grid, represent it explicitly with `LineSlot` only.
* Build grid as two line families:

  * vertical: `slot.grid.v1`, `slot.grid.v2`, ...
  * horizontal: `slot.grid.h1`, `slot.grid.h2`, ...
* Use stable styling for school worksheet grids:

  * `stroke="#9AA0A6"`
  * `stroke_width=1.2`
  * `stroke_dasharray="5 3"`
* Keep grid spacing uniform.
* Extend width/height by whole-cell increments only.

## 6-2) Point-on-Grid Rule

* Plot marked points with `CircleSlot(cx=..., cy=..., r=...)`.
* Never use legacy `CircleSlot(x=..., y=...)`.
* Recommended defaults:

  * `r` around `3.5 ~ 4.0`
  * `fill="#222222"`
* Korean point labels such as `ㄱ`, `ㄴ`, `ㄷ`, `ㄹ`, `ㅁ` should be separate `TextSlot`s.
* Do not merge point labels into point ids.

## 6-3) Slot Signature Guard

* `style_role` is valid only for `TextSlot`.
* For non-text slots, do not emit `style_role`.
* This applies to:

  * `RectSlot`
  * `LineSlot`
  * `CircleSlot`
  * `PolygonSlot`
  * `PathSlot`

[원본에 보이는 요소]

  - `□ 78. 원의 반지름과 지름을 나타내는 선분을 각각 찾아 기호를 선택해 보세요.`

- 그림 아래 선택 문장:

- 좌하단 표기:

- **추론되는 수학적 의미:** 정답과 해설이 별도 표시로 제시된 구성처럼 보인다.

[수식/관계 구조]
- 본 문항에는 수식, 계산식, 등호(`=`)로 이어지는 관계식이 보이지 않는다.

  - `원의 반지름` ↔ `( ㄱ , ㄴ , ㄷ )` 중 선택
  - `원의 지름` ↔ `( ㄱ , ㄴ , ㄷ )` 중 선택
- **보이는 사실:** 원 안에 여러 선분이 겹쳐 그려져 있고, 각 선분에 기호가 붙어 있다.

- 해당 없음.

[빈칸 정책]
- 본 화면에는 전형적인 숫자 빈칸은 보이지 않지만, 선택지 괄호 `( ㄱ , ㄴ , ㄷ )`가 **선택 대상 자리**처럼 기능한다.
- **빈칸 정책:** 추론값으로 채우지 말고, 원본에 보이는 괄호와 선택지 구조를 그대로 유지한다.
- 원문에 보이는 `(정답) ㄷ, ㄴ`, `(해설) 원의 반지름 ㄷ, 원의 지름 ㄴ`은 **관찰 사실로만 기록**하고, 이를 근거로 빈칸을 재작성하지 않는다.
- 입력 칸처럼 보이는 별도 박스는 없으므로, DSL에서는 선택 문장과 해설 표기를 분리된 텍스트 요소로 다루는 편이 좋다.

[반복 패턴]
- 원 안의 도형 후보가 여러 개 반복적으로 배치되어 있다.
- 기호 `ㄱ`, `ㄴ`, `ㄷ`가 각각 선분 후보 옆에 붙어 있어, **기호-도형 매칭 패턴**이 반복된다.
- 선택 문장에서는 같은 형식이 반복된다.
  - `원의 반지름 ( ... )`
  - `원의 지름 ( ... )`
- 좌하단의 `(정답)`, `(해설)`도 동일한 안내 형식으로 반복 배치되어 있다.
- 원 내부에는 파란색 계열의 점선 강조가 여러 개 보이며, 서로 다른 선분 후보를 구분하는 패턴처럼 보인다.

[DSL 작성 힌트]
- 전체 레이아웃은 다음처럼 분할하면 자연스럽다.
  - 상단 좌측: 문항 번호 + 지시문
  - 중앙 우측: 원 도형
  - 원 아래: 선택 문장
  - 좌하단: `(정답)`, `(해설)` 텍스트
- 원 도형 구성 요소:
  - 바깥 원 1개
  - 내부 선분 후보 3개 정도
  - 중심점 1개
  - 각 선분 옆 기호 라벨 `ㄱ`, `ㄴ`, `ㄷ`
- **렌더링/레이아웃 힌트**
  - 원은 검은 외곽선으로, 내부 후보 선분은 연한 파란색 점선처럼 보이게 표현하는 것이 원본에 가깝다.
  - 한 선분은 원 내부를 거의 세로로 내려오고, 한 선분은 비스듬히 지나가며, 한 선분은 중심을 지나가는 긴 선처럼 보인다.
  - 중심 부근의 작은 점과 문자 `O`는 서로 가까이 배치되어 있으므로 겹침 가능성을 고려해 구성한다.
  - 선택 문장은 한 줄에 길게 놓이고, 괄호와 쉼표 간격을 원본처럼 유지하는 것이 좋다.
- `(정답)`, `(해설)`은 하단 좌측에 작은 텍스트 박스로 따로 두는 배치가 적절하다.
- **보이는 사실:** 원문에는 출력 예시처럼 보이는 정답/해설 텍스트가 함께 있다.
- **추론되는 수학적 의미:** 학습자용 풀이 공간이 아니라 해설 포함형 화면일 가능성이 있다.

- 문제는 **원의 반지름과 지름의 정의를 시각적으로 판별**하는 유형으로 보인다.
