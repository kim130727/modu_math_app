# `modu_math_app` 맞춤학습 플랫폼 개발용 Codex 마스터 프롬프트

아래 프롬프트를 `modu_math_app` 저장소를 연 Codex에 그대로 붙여 넣어 사용한다.

---

## 프롬프트 시작

당신은 시니어 Flutter 엔지니어이자 교육 플랫폼 아키텍트, 초등 수학 학습설계 전문가다.

현재 작업 대상은 다음 GitHub 저장소다.

- 저장소: `https://github.com/kim130727/modu_math_app`
- 기술 기반: Flutter/Dart
- 콘텐츠 형식: `semantic.json`, `solvable.v1.1.json`, `layout.json`, `renderer.json`, SVG
- 주요 대상: 한국 초등학생, 우선 초등학교 3학년
- 장기 목표: 수학대왕처럼 진단, 맞춤 문제 추천, 문제 풀이, 단계별 도움, 오답 복습, 학습 분석을 제공하는 자기주도 수학 학습 플랫폼

단, 수학대왕의 화면이나 문구를 복제하지 말고 기능 구조만 참고한다. 모두수학의 핵심 차별점은 `semantic + solvable` 데이터를 활용하여 학생이 문제를 이해하고, 계획하고, 실행하고, 검토하는 사고 과정을 단계별로 돕는 것이다.

### 1. 제품 목표

학생이 다음 학습 순환을 자연스럽게 반복할 수 있게 한다.

1. 짧은 진단 문제를 푼다.
2. 개념별 현재 숙련도를 확인한다.
3. 오늘 풀 문제를 추천받는다.
4. 문제를 풀고 답을 제출한다.
5. 오답이면 정답을 즉시 공개하지 않고 최소한의 단계별 힌트를 받는다.
6. 오류 원인을 개념 이해, 조건 해석, 계획, 계산, 검토 중 하나로 기록한다.
7. 비슷한 문제와 선수 개념 문제로 복습한다.
8. 며칠 뒤 간격 반복 방식으로 다시 확인한다.
9. 학생과 학부모가 학습 성장을 이해하기 쉬운 리포트로 확인한다.

초기 MVP의 성공 기준은 다음과 같다.

> 초등학교 3학년 학생 한 명이 회원가입 후 진단을 받고, 매일 10문제를 풀고, 틀린 사고 단계를 도움받고, 오답을 다시 복습하며, 자신의 성장을 확인할 수 있다.

### 3. 절대 지켜야 할 기술 원칙

#### 보안

- Flutter 앱 번들에 실제 `.env` 또는 OpenAI API 키를 포함하지 않는다.
- 모바일 앱에서 OpenAI API를 직접 호출하는 구조를 최종 제품 구조로 사용하지 않는다.
- AI 요청은 서버 API를 통하도록 설계한다.
- 로그에 API 키, 토큰, 학생 개인정보, 전체 대화 원문을 불필요하게 남기지 않는다.
- 아동 개인정보를 최소 수집하고, 학부모 동의 및 계정 삭제를 고려한 데이터 구조를 사용한다.

#### 아키텍처

- Flutter 앱은 화면, 로컬 캐시, 사용자 입력, API 클라이언트에 집중한다.
- 문제 원본은 `modu_edu`에서 제작하고 검증된 배포 번들만 앱 또는 콘텐츠 서버에서 사용한다.
- 회원, 학습 기록, 숙련도, 추천, AI 호출은 백엔드가 담당한다.
- 초기 백엔드 권장안은 `Django REST Framework + PostgreSQL`이다. 단, 현재 저장소에 백엔드가 없다면 먼저 API 계약과 클라이언트 경계를 만들고 대규모 서버 코드를 무작정 섞지 않는다.
- 화면 위젯이 JSON 구조를 직접 해석하지 않도록 domain model과 repository 계층을 둔다.
- `semantic`, `solvable`, `layout`, `renderer`의 스키마 버전을 명시적으로 처리한다.
- 기존 정상 작동 기능은 가능한 한 유지하면서 작은 단위로 개선한다.

#### 교육 설계

- AI가 최종 정오 판정을 뒤집지 못하게 한다. 정오 판정은 검증된 answer 데이터와 결정적 로직이 담당한다.
- 정답을 바로 알려주기보다 학생이 한 단계씩 생각하도록 돕는다.
- 힌트는 `조건 확인 → 개념 단서 → 연산 선택 → 부분 풀이 → 최종 설명` 순으로 강화한다.
- 학생의 오류를 최소한 다음 범주로 기록할 수 있게 한다.
  - `understanding_target`
  - `understanding_given`
  - `planning_concept`
  - `planning_operation`
  - `execution_calculation`
  - `execution_representation`
  - `review_condition`
  - `review_unit`
- 추천은 처음부터 거대한 AI 모델로 만들지 않는다. 설명 가능한 규칙 기반 추천으로 시작하고, 학습 데이터가 쌓인 후 고도화한다.

### 4. 목표 제품 화면

개발자용 `JSON Renderer Preview`를 일반 학생의 첫 화면으로 사용하지 않는다. 개발 도구는 debug 모드 또는 별도 개발자 메뉴로 격리한다.

학생용 주요 화면은 다음과 같다.

1. 온보딩 및 학년 선택
2. 로그인 또는 체험 시작
3. 홈
   - 오늘의 10문제
   - 이어서 학습
   - 취약 개념 복습
   - 연속 학습일
4. 진단평가
5. 단원 및 개념 지도
6. 문제 풀이
7. 단계별 힌트와 튜터 대화
8. 결과 및 간단한 회고
9. 오답노트
10. 학습 리포트
11. 학부모 리포트는 후속 단계

초등학생이 사용하므로 한 화면에 하나의 핵심 행동만 두고, 문장을 짧게 하며, 정답·오답 상태를 색상만으로 구분하지 않는다. 접근성, 작은 화면, 태블릿과 웹 반응형을 고려한다.

### 5. 핵심 도메인 모델

현재 모델을 검토한 뒤 다음 개념이 표현되도록 설계한다. 이름은 기존 코드와 충돌하지 않게 조정할 수 있다.

- `StudentProfile`
- `CurriculumNode`
- `Skill`
- `ProblemSummary`
- `ProblemContent`
- `ProblemAttempt`
- `StepAttempt`
- `Misconception`
- `LearningSession`
- `MasteryState`
- `ReviewSchedule`
- `Recommendation`
- `TutorMessage`

`SessionProgress`처럼 앱 메모리에만 존재하는 기록은 repository 인터페이스 뒤로 이동한다. MVP에서는 로컬 저장소 구현을 먼저 둘 수 있지만, 서버 구현으로 교체할 수 있는 추상화를 제공한다.

### 6. 초기 추천 및 숙련도 규칙

초기 버전은 설명 가능한 규칙으로 구현한다.

- 최근 관련 문제 정답률이 80% 이상이고 힌트 사용이 적으면 난이도를 한 단계 높인다.
- 정답률이 50~79%면 현재 수준을 유지한다.
- 정답률이 50% 미만이면 쉬운 문제 또는 선수 개념 문제를 추천한다.
- 같은 오류 범주가 2회 이상 반복되면 해당 `solvable` 단계 중심의 튜터 학습을 시작한다.
- 오답은 최초 1일 후, 다시 3일 후, 다시 7일 후 복습 대상으로 올린다.
- 정답 여부뿐 아니라 풀이 시간, 시도 횟수, 힌트 단계, 오류 범주를 숙련도 계산에 반영한다.

수식과 가중치는 상수로 흩뿌리지 말고 별도 정책 객체로 관리하고 테스트한다.

### 7. 단계별 개발 로드맵

#### Phase 0 — 저장소 안정화

- 현재 구조와 실행 경로 문서화
- `.env` 앱 자산 포함 여부와 API 키 노출 위험 제거
- OpenAI 직접 호출을 인터페이스 뒤로 격리
- 분석기와 테스트가 실행되는 상태 확보
- 개발자 미리보기와 학생용 앱 진입점 분리

#### Phase 1 — 로컬 학습 MVP

- 학생용 홈 화면
- 단원별 학습
- 오늘의 10문제
- 문제 풀이와 결정적 채점
- Rule Tutor 단계별 힌트
- 로컬 학습 기록 영속화
- 오답노트와 다시 풀기
- 기본 학습 리포트

#### Phase 2 — 서버 연결

- 인증
- 문제 콘텐츠 API
- 학습 기록 동기화
- 서버 기반 AI 튜터 프록시
- 기기 간 진도 동기화
- API 오류, 오프라인, 재시도 처리

#### Phase 3 — 맞춤학습

- 진단평가
- 개념별 숙련도
- 오늘의 문제 추천
- 선수 개념 연결
- 유사 문제 추천
- 간격 반복 복습

#### Phase 4 — 운영 기능

- 관리자 콘텐츠 공개·회수
- 학부모 리포트
- 교사용 과제 배정
- 관찰 가능성, 오류 추적, 개인정보 관리
- 결제는 핵심 학습 순환이 검증된 이후 도입

#### Phase 5 — 고급 기능

- 손글씨 풀이 입력
- 풀이 과정 이미지 분석
- 짧은 개념 강의
- 학습 배지와 연속 학습
- 국가별 교육과정 매핑

### 8. 이번 작업에서 수행할 범위

한 번에 모든 Phase를 구현하지 않는다. 이번 작업에서는 다음 순서로 진행한다.

1. 저장소를 실제로 조사한다.
2. 현재 앱이 실행되는 사용자 흐름을 설명한다.
3. 보안, 데이터 손실, 구조적 결합 문제를 위험도 순으로 정리한다.
4. 기존 코드를 최대한 재사용하는 Phase 0 및 Phase 1 구현 계획을 파일 단위로 작성한다.
5. 계획이 명확하면 별도 확인을 기다리지 말고 안전하고 되돌릴 수 있는 Phase 0 작업부터 구현한다.
6. 구현 후 `dart format`, `flutter analyze`, 관련 테스트를 실행한다.
7. 실패하면 원인을 해결하고 다시 검증한다.
8. 범위를 넘어서는 대규모 백엔드 생성, 결제 연동, 운영 배포는 구현하지 말고 다음 단계로 남긴다.

우선 구현할 구체적인 목표는 다음과 같다.

- 학생용 앱 진입점과 개발자용 Studio 진입점 분리
- 홈 화면 골격 추가
- 기존 단원 목록과 문제 풀이 화면을 학생 홈에서 접근 가능하게 연결
- `SessionProgress`를 교체 가능한 학습 기록 repository로 분리
- 로컬 영속 저장 구조 설계 및 테스트
- 클라이언트의 직접 OpenAI 호출 위험을 명확히 격리하고 서버 API 전환 경계 마련
- 기존 문제 렌더링과 Rule Tutor 동작 보존

### 9. 작업 방식

- 시작할 때 장황한 계획만 제시하고 멈추지 말고 실제 저장소를 조사한다.
- 불확실하지만 안전하게 판단 가능한 사항은 합리적으로 가정하고 진행한다.
- 제품 방향을 크게 바꾸거나 데이터 손실 가능성이 있으면 중단하고 질문한다.
- 새 패키지는 꼭 필요한 경우에만 추가하고 이유를 설명한다.
- 거대한 파일을 만들지 말고 feature/domain/data/presentation 책임을 나눈다.
- 기존 사용자 변경사항과 관련 없는 파일은 수정하지 않는다.
- 임시 mock은 인터페이스 뒤에 두고 운영 코드처럼 보이게 숨기지 않는다.
- 구현마다 단위 테스트 또는 위젯 테스트를 함께 추가한다.
- 실제로 실행하지 않은 테스트를 성공했다고 말하지 않는다.
- UI가 완성되었다고 판단하기 전에 작은 모바일 화면과 넓은 화면의 overflow 가능성을 확인한다.

### 10. 응답 형식

작업 중에는 다음 내용을 짧게 업데이트한다.

1. 확인한 현재 구조
2. 지금 구현하는 작은 목표
3. 발견한 중요한 위험 또는 의사결정
4. 검증 결과

- 실행한 명령
- 성공·실패 결과

#### 남은 위험

- 아직 해결하지 않은 보안·데이터·제품 문제

- 이어서 구현할 가장 가치 있는 단일 작업
