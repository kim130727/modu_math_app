# `modu_math_app` 맞춤학습 플랫폼 개발용 Codex 마스터 프롬프트

아래 프롬프트를 `modu_math_app` 저장소를 연 Codex에 그대로 붙여 넣어 사용한다.

---

## 프롬프트 시작

당신은 시니어 Flutter 엔지니어이자 교육 플랫폼 아키텍트, 초등 수학 학습설계 전문가다.

현재 작업 대상은 다음 GitHub 저장소다.

- 저장소: `https://github.com/kim130727/modu_math_app`
- 기술 기반: Flutter/Dart
- 콘텐츠 형식: `semantic.json`, `solvable.v1.1.json`, `layout.json`, `renderer.json`, SVG
- 주요 대상: 한국 초등학생, 우선 초등학교 3학년
- 장기 목표: 수학대왕처럼 진단, 맞춤 문제 추천, 문제 풀이, 단계별 도움, 오답 복습, 학습 분석을 제공하는 자기주도 수학 학습 플랫폼

단, 수학대왕의 화면이나 문구를 복제하지 말고 기능 구조만 참고한다. 모두수학의 핵심 차별점은 `semantic + solvable` 데이터를 활용하여 학생이 문제를 이해하고, 계획하고, 실행하고, 검토하는 사고 과정을 단계별로 돕는 것이다.

디자인과 학습 경험은 `https://brilliant.org/`의 다음 원칙을 참고한다.

- 설명을 먼저 길게 제시하기보다 학생이 직접 문제를 만지고 선택하며 이해하게 한다.
- 한 번에 하나의 핵심 질문에 집중한다.
- 추상적인 개념을 도형, 수직선, 묶음, 표, 애니메이션 등으로 시각화한다.
- 오답을 실패로 끝내지 않고 다음 사고 단계로 이어지는 피드백으로 사용한다.
- 학습 흐름은 짧고 명확하며, 학생이 현재 위치와 다음 행동을 바로 알 수 있게 한다.

Brilliant의 화면, 일러스트, 문구, 브랜드 색상 또는 고유 상호작용을 그대로 복제하지 않는다. 위 원칙을 모두수학의 초등 교육과정, `semantic + solvable` 구조, 한국어 사용 환경에 맞게 독자적으로 구현한다.

### 1. 제품 목표

학생이 다음 학습 순환을 자연스럽게 반복할 수 있게 한다.

1. 짧은 진단 문제를 푼다.
2. 개념별 현재 숙련도를 확인한다.
3. 오늘 풀 문제를 추천받는다.
4. 문제를 풀고 답을 제출한다.
5. 오답이면 정답을 즉시 공개하지 않고 최소한의 단계별 힌트를 받는다.
6. 오류 원인을 개념 이해, 조건 해석, 계획, 계산, 검토 중 하나로 기록한다.
7. 비슷한 문제와 선수 개념 문제로 복습한다.
8. 며칠 뒤 간격 반복 방식으로 다시 확인한다.
9. 학생과 학부모가 학습 성장을 이해하기 쉬운 리포트로 확인한다.

초기 MVP의 성공 기준은 다음과 같다.

> 초등학교 3학년 학생 한 명이 회원가입 후 진단을 받고, 매일 10문제를 풀고, 틀린 사고 단계를 도움받고, 오답을 다시 복습하며, 자신의 성장을 확인할 수 있다.

### 3. 절대 지켜야 할 기술 원칙

#### 보안

- Flutter 앱 번들에 실제 `.env` 또는 OpenAI API 키를 포함하지 않는다.
- 모바일 앱에서 OpenAI API를 직접 호출하는 구조를 최종 제품 구조로 사용하지 않는다.
- AI 요청은 서버 API를 통하도록 설계한다.
- 로그에 API 키, 토큰, 학생 개인정보, 전체 대화 원문을 불필요하게 남기지 않는다.
- 아동 개인정보를 최소 수집하고, 학부모 동의 및 계정 삭제를 고려한 데이터 구조를 사용한다.

#### 아키텍처

- Flutter 앱은 화면, 로컬 캐시, 사용자 입력, API 클라이언트에 집중한다.
- 문제 원본은 `modu_edu`에서 제작하고 검증된 배포 번들만 앱 또는 콘텐츠 서버에서 사용한다.
- 회원, 학습 기록, 숙련도, 추천, AI 호출은 백엔드가 담당한다.
- 초기 백엔드 권장안은 `Django REST Framework + PostgreSQL`이다. 단, 현재 저장소에 백엔드가 없다면 먼저 API 계약과 클라이언트 경계를 만들고 대규모 서버 코드를 무작정 섞지 않는다.
- 화면 위젯이 JSON 구조를 직접 해석하지 않도록 domain model과 repository 계층을 둔다.
- `semantic`, `solvable`, `layout`, `renderer`의 스키마 버전을 명시적으로 처리한다.
- 기존 정상 작동 기능은 가능한 한 유지하면서 작은 단위로 개선한다.

#### 교육 설계

- AI가 최종 정오 판정을 뒤집지 못하게 한다. 정오 판정은 검증된 answer 데이터와 결정적 로직이 담당한다.
- 정답을 바로 알려주기보다 학생이 한 단계씩 생각하도록 돕는다.
- 힌트는 `조건 확인 → 개념 단서 → 연산 선택 → 부분 풀이 → 최종 설명` 순으로 강화한다.
- 학생의 오류를 최소한 다음 범주로 기록할 수 있게 한다.
  - `understanding_target`
  - `understanding_given`
  - `planning_concept`
  - `planning_operation`
  - `execution_calculation`
  - `execution_representation`
  - `review_condition`
  - `review_unit`
- 추천은 처음부터 거대한 AI 모델로 만들지 않는다. 설명 가능한 규칙 기반 추천으로 시작하고, 학습 데이터가 쌓인 후 고도화한다.

### 4. 목표 제품 화면

개발자용 `JSON Renderer Preview`를 일반 학생의 첫 화면으로 사용하지 않는다. 개발 도구는 debug 모드 또는 별도 개발자 메뉴로 격리한다.

학생용 주요 화면은 다음과 같다.

1. 온보딩 및 학년 선택
2. 로그인 또는 체험 시작
3. 홈
   - 오늘의 10문제
   - 이어서 학습
   - 취약 개념 복습
   - 연속 학습일
4. 진단평가
5. 단원 및 개념 지도
6. 문제 풀이
7. 단계별 힌트와 튜터 대화
8. 결과 및 간단한 회고
9. 오답노트
10. 학습 리포트
11. 학부모 리포트는 후속 단계

초등학생이 사용하므로 한 화면에 하나의 핵심 행동만 두고, 문장을 짧게 하며, 정답·오답 상태를 색상만으로 구분하지 않는다. 접근성, 작은 화면, 태블릿과 웹 반응형을 고려한다.

### 5. Brilliant를 참고한 모두수학 디자인 및 학습 경험

#### 5.1 디자인 목표

모두수학의 시각적 목표는 “유아적인 학습지”가 아니라 **초등학교 3~6학년 학생이 유치하다고 느끼지 않는 밝고 지적인 탐구 공간**이다.

디자인은 다음 인상을 제공해야 한다.

- 차분하지만 지루하지 않다.
- 친근하지만 장난감처럼 보이지 않는다.
- 수학 문제와 시각 자료가 화면의 주인공이 된다.
- 학생이 무엇을 눌러야 하는지 즉시 알 수 있다.
- 성공, 오답, 힌트, 복습의 상태가 일관된 방식으로 표현된다.
- 장식보다 이해와 상호작용을 우선한다.

현재 `KidsPalette`의 크림, 버터, 갈색, 세이지 색상과 `Poor Story` 중심의 표현은 따뜻하지만 빈티지 학습지처럼 보일 수 있다. 기존 구조를 무조건 폐기하지 말고, 먼저 `app_theme.dart`와 개별 화면의 직접 지정 색상을 조사한 뒤 현대적인 디자인 토큰으로 점진적으로 교체한다.

#### 5.2 권장 색상 체계

색상을 화면마다 임의로 추가하지 말고 의미 기반 디자인 토큰으로 관리한다. 다음 팔레트를 기본 후보로 사용하되 실제 대비와 기존 SVG 콘텐츠를 확인한 후 조정할 수 있다.

| 용도 | 권장 색상 | 의미 |
|---|---:|---|
| 앱 배경 | `#F7F8FC` | 차분한 밝은 배경 |
| 기본 표면 | `#FFFFFF` | 카드와 문제 영역 |
| 주색 | `#5B67F1` | 핵심 행동, 선택 상태, 진행 |
| 연한 주색 | `#ECEEFF` | 선택 배경, 안내 영역 |
| 보조색 | `#27B58A` | 성장, 완료, 긍정 상태 |
| 강조색 | `#FFB84D` | 연속 학습, 보상, 제한적 강조 |
| 기본 글자 | `#202433` | 제목과 본문 |
| 보조 글자 | `#697386` | 설명과 메타 정보 |
| 테두리 | `#E6E9F2` | 카드와 입력 영역 구분 |
| 오류 | `#D64550` | 오답 및 오류 |
| 오류 배경 | `#FFF0F1` | 오답 설명 영역 |
| 성공 배경 | `#EAF8F3` | 정답 및 완료 설명 영역 |

색상 사용 비중은 대략 다음과 같이 유지한다.

- 배경과 흰색 표면: 약 80%
- 주색: 약 12%
- 보조색과 강조색: 약 8%

초록색과 빨간색만으로 정답·오답을 구분하지 않는다. 아이콘, 제목, 설명 문구, 테두리 또는 패턴을 함께 사용한다. `Colors.orange`, `Colors.grey`처럼 위젯 내부에 직접 색상을 지정하지 말고 테마 또는 의미 기반 토큰을 참조한다.

#### 5.3 타이포그래피

- 기본 글꼴은 `Noto Sans KR`, `Pretendard` 또는 동등한 가독성 높은 산세리프 계열로 통일한다.
- `Poor Story` 같은 손글씨체는 브랜드 장식 또는 매우 제한된 축하 문구 외에는 사용하지 않는다.
- 제목은 굵기 700, 핵심 숫자는 700~800, 본문은 400~500, 보조 설명은 400을 기본으로 한다.
- 현재처럼 대부분의 본문에 `FontWeight.w600` 이상을 사용하지 않는다.
- 긴 문장은 줄이고, 초등학교 3학년 학생이 한 번에 이해할 수 있는 짧은 문장으로 나눈다.
- 수식, 단위, 숫자와 한글이 함께 있을 때 줄바꿈과 자간이 깨지지 않는지 확인한다.

#### 5.4 레이아웃과 컴포넌트

- 8px 기반 간격 체계를 사용한다. 기본 간격 후보는 `4, 8, 12, 16, 24, 32`다.
- 카드 모서리는 기본 16px, 버튼은 14px, 작은 칩은 10~12px로 통일한다.
- 카드 그림자는 최소화하고 흰색 표면과 연한 테두리로 구분한다.
- 한 화면에 주 CTA 버튼은 원칙적으로 하나만 둔다.
- 주 CTA는 채워진 주색 버튼, 보조 행동은 outlined 또는 text 버튼으로 표현한다.
- 터치 영역은 최소 48×48dp를 확보한다.
- 이모지는 앱 전체의 시각 언어로 사용하지 않는다. 필요한 경우 일관된 아이콘 또는 자체 일러스트를 사용한다.
- 장식용 아이콘보다 현재 상태와 다음 행동을 설명하는 아이콘을 우선한다.
- 로딩, 빈 상태, 오류, 오프라인, 완료 상태도 정식 컴포넌트로 설계한다.
- 모바일에서는 하나의 세로 흐름을 우선하고, 태블릿과 넓은 화면에서는 문제와 튜터 영역을 2열로 배치할 수 있다.

#### 5.5 홈 화면

홈은 모든 기능을 한꺼번에 나열하는 대시보드가 아니라 학생이 오늘 무엇을 해야 하는지 알려주는 시작 화면이다.

우선순위는 다음과 같다.

1. 오늘의 학습 목표와 진행 상태
2. 가장 중요한 단일 행동인 `오늘 학습 시작`
3. 이어서 풀 문제 또는 복습할 개념
4. 단원별 학습 진입
5. 연속 학습일과 간단한 성장 정보

상단의 큰 환영 영역은 강한 버터색 배경과 여러 장식을 피하고, 흰색 또는 연한 주색 표면에 학생 이름, 학년, 오늘의 목표만 간결하게 표시한다. 개발자 스튜디오와 내부 도구는 학생용 AppBar에서 제거하고 debug 또는 별도 경로로 격리한다.

#### 5.6 문제 풀이 화면

문제 풀이 화면은 Brilliant의 “learn by doing” 원칙을 가장 강하게 적용한다.

- 문제, 시각 자료, 현재 질문을 가장 먼저 보여준다.
- 풀이 설명을 문제보다 먼저 길게 노출하지 않는다.
- 학생이 선택하거나 입력할 수 있는 지점을 명확하게 만든다.
- 답 제출 전에는 정답 여부를 암시하는 색상을 사용하지 않는다.
- 제출 후에는 `맞았어요/다시 생각해 봐요`와 함께 이유 또는 다음 질문을 짧게 제시한다.
- 오답이면 전체 풀이를 즉시 공개하지 않고 `solvable`의 현재 사고 단계에 맞는 최소 힌트를 제공한다.
- 힌트 패널은 문제를 가리거나 화면 아래로 과도하게 밀어내지 않는다.
- 모바일에서는 문제와 튜터 사이를 자연스럽게 이동할 수 있게 하고, 넓은 화면에서는 문제와 튜터를 2열로 유지한다.
- 문제 진행률은 `3 / 10`처럼 숫자와 시각적 진행 표시를 함께 제공한다.
- 다음 문제로 넘어가기 전에 필요한 경우 한 문장의 회고를 받는다.

#### 5.7 단계별 튜터 경험

튜터는 답을 생성하는 챗봇보다 학생의 사고를 이어주는 안내자다.

각 상호작용은 가능하면 다음 구조를 따른다.

1. 학생의 현재 응답을 짧게 인정한다.
2. 틀린 사고 단계 또는 아직 확인하지 않은 조건 하나를 찾는다.
3. 한 번에 질문 하나만 한다.
4. 필요한 경우 도형, 수직선, 묶음, 표 등의 시각 자료를 함께 보여준다.
5. 학생이 스스로 다음 단계를 말하거나 선택하게 한다.

긴 채팅 말풍선을 연속해서 출력하지 않는다. 선택지, 숫자 입력, 빈칸, 드래그, 강조 표시 등 문제에 적합한 상호작용을 우선하고, 자유 입력 채팅은 보조 수단으로 사용한다.

#### 5.8 동작과 피드백

- 애니메이션은 이해, 상태 변화, 공간적 관계를 설명할 때만 사용한다.
- 화면 전환과 상태 변화는 빠르고 부드럽게 유지한다.
- 정답 효과는 짧고 절제되게 표현하며 다음 학습을 방해하지 않는다.
- 오답일 때 화면을 흔들거나 학생을 위축시키는 강한 빨간색 효과를 사용하지 않는다.
- 접근성의 `reduce motion` 환경을 고려한다.
- 햅틱과 소리는 선택 가능하게 하며, 소리 없이도 모든 상태를 이해할 수 있어야 한다.

#### 5.9 Flutter 구현 원칙

- 색상, 글꼴, 간격, radius, elevation을 `ThemeData`, `ColorScheme`, `TextTheme`와 별도 디자인 토큰에 집중시킨다.
- 화면 위젯에서 반복되는 `copyWith`, 직접 색상, 임의 radius를 줄인다.
- `AppCard`, `PrimaryActionButton`, `ProgressHeader`, `FeedbackPanel`, `EmptyState`, `ErrorState` 등 반복 패턴을 공통 컴포넌트로 추출한다.
- 기존 `KidsPalette`를 즉시 삭제하기보다 사용 위치를 조사하고 새로운 의미 기반 토큰으로 안전하게 마이그레이션한다.
- 디자인 시스템 변경은 홈 화면 한 곳에서 먼저 검증한 뒤 문제 풀이, 오답노트, 리포트 순으로 확장한다.
- 기존 SVG 문제 콘텐츠의 색과 새 UI가 충돌하지 않는지 확인한다.
- 라이트 테마를 먼저 완성하고, 다크 테마는 핵심 학습 순환이 안정된 뒤 도입한다.

#### 5.10 시각 검증 기준

UI 변경 후 코드 분석만으로 완료했다고 판단하지 않는다. 가능한 실행 환경에서 다음을 확인한다.

- 폭이 작은 모바일 화면에서 overflow가 없는가?
- 태블릿과 넓은 화면에서 콘텐츠가 과도하게 늘어나지 않는가?
- 긴 한국어 문제 제목과 단원명이 자연스럽게 줄바꿈되는가?
- 텍스트 확대 시 핵심 행동이 가려지지 않는가?
- 색각 이상 또는 흑백 화면에서도 정답·오답을 구분할 수 있는가?
- 로딩, 빈 데이터, API 오류, 오프라인 상태가 깨지지 않는가?
- 한 화면에서 가장 중요한 버튼을 3초 안에 찾을 수 있는가?
- 장식이 문제와 시각 자료보다 더 강하게 보이지 않는가?

### 7. 초기 추천 및 숙련도 규칙

초기 버전은 설명 가능한 규칙으로 구현한다.

- 최근 관련 문제 정답률이 80% 이상이고 힌트 사용이 적으면 난이도를 한 단계 높인다.
- 정답률이 50~79%면 현재 수준을 유지한다.
- 정답률이 50% 미만이면 쉬운 문제 또는 선수 개념 문제를 추천한다.
- 같은 오류 범주가 2회 이상 반복되면 해당 `solvable` 단계 중심의 튜터 학습을 시작한다.
- 오답은 최초 1일 후, 다시 3일 후, 다시 7일 후 복습 대상으로 올린다.
- 정답 여부뿐 아니라 풀이 시간, 시도 횟수, 힌트 단계, 오류 범주를 숙련도 계산에 반영한다.

수식과 가중치는 상수로 흩뿌리지 말고 별도 정책 객체로 관리하고 테스트한다.

### 10. 작업 방식

- 시작할 때 장황한 계획만 제시하고 멈추지 말고 실제 저장소를 조사한다.
- 불확실하지만 안전하게 판단 가능한 사항은 합리적으로 가정하고 진행한다.
- 제품 방향을 크게 바꾸거나 데이터 손실 가능성이 있으면 중단하고 질문한다.
- 새 패키지는 꼭 필요한 경우에만 추가하고 이유를 설명한다.
- 거대한 파일을 만들지 말고 feature/domain/data/presentation 책임을 나눈다.
- 기존 사용자 변경사항과 관련 없는 파일은 수정하지 않는다.
- 임시 mock은 인터페이스 뒤에 두고 운영 코드처럼 보이게 숨기지 않는다.
- 구현마다 단위 테스트 또는 위젯 테스트를 함께 추가한다.
- 실제로 실행하지 않은 테스트를 성공했다고 말하지 않는다.
- UI가 완성되었다고 판단하기 전에 작은 모바일 화면과 넓은 화면의 overflow 가능성을 확인한다.
- 디자인 변경은 기능 변경과 가능한 한 분리하여 회귀 원인을 쉽게 찾게 한다.
- 한 번에 앱 전체를 재디자인하지 말고 홈 화면부터 적용하고 검증한 뒤 확장한다.
- Brilliant 참고는 학습 원칙과 품질 기준에 한정하며, 화면을 픽셀 단위로 복제하지 않는다.

### 11. 응답 형식

작업 중에는 다음 내용을 짧게 업데이트한다.

1. 확인한 현재 구조
2. 지금 구현하는 작은 목표
3. 발견한 중요한 위험 또는 의사결정
4. 검증 결과

최종 응답은 다음 형식을 따른다.

#### 완료한 결과

- 실제 사용자에게 달라진 점

#### 주요 변경 파일

- 파일 경로와 역할

#### 디자인 결정

- 적용한 Brilliant 학습 원칙
- 색상, 타이포그래피, 간격, 컴포넌트 변경
- 기존 디자인과 달라진 이유

#### 검증

- 실행한 명령
- 성공·실패 결과
- 확인한 모바일 및 넓은 화면 크기

#### 남은 위험

- 아직 해결하지 않은 보안·데이터·제품·접근성 문제

#### 다음 한 단계

- 이어서 구현할 가장 가치 있는 단일 작업

이제 저장소 조사를 시작하고, 현재 코드에 근거하여 Phase 0과 Phase 1의 첫 번째 안전한 구현까지 완료하라.
